# 07-v1-plan-wizard Specification

## Purpose

Provide card-based create-plan screens that guide the user through defining their training
requirements and produce a typed `PlanSpec` data structure consumed by other capabilities
(specifically, change 08 — AI plan generation).

The card-mode stepper flow is the v1 create-plan scope. Conversational create-plan and
voice-assistant screens are delivered by v1.1 specs. This change (07) produces a confirmed
`PlanSpec` (the user's training *requirements*) ONLY. Generating the actual workout program
(exercises, sets, reps, rest, weekly schedule) from a `PlanSpec` is the exclusive
responsibility of change 08 (AI plan generation).

## Requirements

### Requirement: Wizard Step Progression

The wizard MUST present six sequential steps in this order: goal, location, frequency, duration,
equipment, limitations. `location` MUST precede `equipment` because the selected location
constrains which equipment options are available. The wizard MAY allow backward navigation;
Back navigation MUST preserve the values already entered for all prior steps.

A plan MUST NOT be confirmable unless every required step has been completed. Goal, location,
frequency, and duration MUST each carry a non-null value. Equipment and limitations MUST each
have their step visited and submitted, but MAY resolve to an empty array.

#### Scenario: Complete wizard flow

- GIVEN a user on the wizard start screen
- WHEN they complete all six steps and click "Finish"
- THEN a confirmed `PlanSpec` object (the user's training requirements) is persisted to the
  database with all six fields populated, AND no workout program (exercises/sets/schedule) is
  generated by this wizard — that is the responsibility of change 08

#### Scenario: Back navigation preserves state

- GIVEN a user on step 5 (equipment)
- WHEN they click "Back" to step 4 (duration)
- THEN previously entered values for steps 1–4 are preserved

#### Scenario: Confirm blocked when required step incomplete

- GIVEN a user who has completed goal and location but skipped frequency
- WHEN they attempt to reach the Finish action
- THEN the wizard MUST NOT allow submission and MUST indicate which required step is incomplete

#### Scenario: Equipment step visited with empty selection

- GIVEN a user who has no equipment available
- WHEN they visit the equipment step and submit with an empty selection
- THEN the step is considered complete and the wizard advances to limitations

### Requirement: PlanSpec Output

The wizard MUST produce a typed `PlanSpec` shared structure containing the user's training
*requirements*: goal, location (home/gym/outdoor), weekly frequency (`daysPerWeek`), session
duration (`sessionDurationMinutes`), available equipment (`string[]`), user limitations
(`{ text: string; isWarning: boolean }[]` — `isWarning` defaults to `true`), and
`preferenceScores` (derived deterministically from wizard answers — not user-entered). On
Finish, this confirmed `PlanSpec` is the persisted output of the wizard.

The `PlanSpec` is the wizard's only persisted output. Generating the actual workout program
(exercises, sets, reps, rest, weekly schedule) from a `PlanSpec` is OUT OF SCOPE and is owned
by change 08 (AI plan generation); the wizard MUST NOT produce that program.

#### Scenario: Limitations flagged as suggestions

- GIVEN a user enters a physical limitation (e.g., "knee pain")
- WHEN the wizard generates a `PlanSpec`
- THEN the limitation is stored as `{ text: "knee pain", isWarning: true }` and no medical
  diagnosis is attempted

#### Scenario: Location captured in output

- GIVEN a user selects "gym" on the location step
- WHEN they complete all remaining steps and click "Finish"
- THEN the persisted `PlanSpec` contains `location: "gym"`

#### Scenario: preferenceScores derived from answers

- GIVEN a user completes the wizard with specific goal, location, frequency, duration,
  and equipment selections
- WHEN the `PlanSpec` is assembled
- THEN `preferenceScores` is computed deterministically from those answers and MUST NOT
  require any additional user input

#### Scenario: Partial wizard exit

- GIVEN a user exits after completing any step
- WHEN they return to the wizard
- THEN they resume at the step immediately after the last completed step, with all previously
  entered data restored from the server-persisted draft

### Requirement: Card-Based Open Design Alignment

The create-plan screens MUST follow the card-based interaction model from the local Open Design
snapshot while preserving the canonical `PlanSpec` output.

#### Scenario: Create-plan screen presents card choices

- GIVEN a user opens the v1 create-plan flow
- WHEN a step renders
- THEN the primary choices are presented as selectable cards rather than conversational prompts

### Requirement: Draft Persistence and Resume

The wizard MUST persist a single active draft per user per tenant on the server after each
step submission. On re-entry, the system MUST resume the user at the step immediately after
the last completed step and restore all prior answers. Completing the final step MUST promote
the draft to a confirmed `PlanSpec` (the wizard's requirements output) — it MUST NOT generate a
workout program. A user who re-enters an existing unfinished draft MUST be offered the option to
continue or overwrite it.

#### Scenario: Draft saved after each step

- GIVEN a user completes step 2 (location) and navigates to step 3
- WHEN the server receives the step submission
- THEN the draft is upserted server-side with the current answers and current step index

#### Scenario: Resume restores exact position

- GIVEN a user previously exited at step 3 (frequency)
- WHEN they navigate back to the create-plan route
- THEN the wizard loads at step 3 with steps 1 and 2 values pre-filled

#### Scenario: Draft promoted to confirmed PlanSpec on Finish

- GIVEN a user completes all six steps
- WHEN they click "Finish"
- THEN the draft is promoted to a confirmed `plan_specs` row (the `PlanSpec`) and the draft
  record is removed, AND no workout program (exercises/sets/schedule) is created — that is 08

#### Scenario: Overwrite prompt on re-entry with existing draft

- GIVEN a user already has an unfinished draft
- WHEN they navigate to the create-plan route
- THEN the system presents the option to continue the existing draft or start over
